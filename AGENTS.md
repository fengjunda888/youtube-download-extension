# Safety Rules

你必须把“避免误删、误改、越权操作”放在第一优先级。

## 删除限制
禁止批量删除文件或目录。
不要使用：
- `del /s`
- `rd /s`
- `rmdir /s`
- `Remove-Item -Recurse`
- `rm -rf`

需要删除文件时，只能一次删除一个明确路径的文件。
正确示例：
`Remove-Item "C:\path\to\file.txt"`

如果需要删除多个文件，必须先停止，并向用户列出将要删除的每个路径，等待用户明确确认。
如果用户没有逐项确认，不得执行。

## 目录范围限制
除非用户明确授权，否则不要对当前项目根目录之外的路径执行读写、移动、重命名、删除、覆盖、清理操作。
禁止把用户主目录、磁盘根目录或系统目录当作工作目录或清理目标，例如：
- `C:\`
- `D:\`
- `C:\Users\`
- `C:\Windows\`
- `C:\Program Files\`

## 高风险操作限制
以下操作默认禁止，除非用户在当前对话中明确批准：
- 批量删除
- 递归清理
- 跨目录移动大量文件
- 覆盖配置文件
- 修改系统环境变量
- 安装或卸载全局软件
- 运行来源不明的脚本
- 联网下载并执行脚本

## 执行前确认
如果操作满足以下任一条件，先停止并向用户确认：
- 会删除文件
- 会覆盖现有文件
- 会修改工作区外内容
- 会运行联网命令
- 会修改系统设置或用户配置
- 会造成不可逆后果

确认时要先给出：
1. 将要操作的准确路径
2. 将要执行的动作
3. 可能的风险

## 安全替代
优先使用安全替代方案：
- 优先“列出将改什么”，再执行修改
- 优先只读分析，后写入
- 优先小范围改动，避免整目录处理
- 优先保留原文件，必要时先备份再改
- 优先项目内相对路径，避免绝对路径

## 工作方式
用户如果没有明确要求你直接执行高风险操作，你应先提出方案，再等待确认。
如果用户表达过对误删的担忧，你必须采用更保守的执行方式。

## 4) “昨天明明推送了但找不到”的常见原因（经验）

- 你记忆中的“昨天 5/27 大量提交并推送”，很可能发生在：
  - 另一个目录下的另一份仓库（`.git` 不同）
  - 另一条 remote（推到了不同仓库/分支）
  - 只改了打包产物（exe/zip），没有提交源码
- 所以排查顺序固定为：
  1) 确认目录是不是同一个 `.git`
  2) 确认当前分支、remote、`git log` 的日期范围
  3) 再谈“回滚/恢复/打包”

## 5) LongRunner 运行策略（避免锁屏/后台中断的思路）

- 计划不要“一次性全发给 Codex”，而是“逐项发送/逐项确认状态”，降低 need-user 时整体卡死概率
- “不打扰/自动继续/跳过”三者的语义边界需要明确：
  - 跳过：无法完成就跳到下一项（但如果后续依赖它，整体仍可能无意义）
  - 自动继续：对非高风险操作自动推进
  - 不打扰：尽可能不弹窗/不抢焦点（尤其锁屏/后台）
- 锁屏导致停止的常见根因不是“锁屏本身”，而是：
  - app-server/子进程异常退出（日志常见 exit code 1）
  - 触发 need-user 决策点且策略为 pause
  - 运行的并不是同一个 exe/同一个配置路径（导致行为不一致）
